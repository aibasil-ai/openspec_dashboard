## Why

目前系統在專案開啟後，若本機檔案被外部更新，畫面不會即時反映，使用者可能在舊資料上操作並做出錯誤判斷。加入即時更新機制可確保畫面與檔案狀態一致，降低資訊延遲造成的操作風險。

## What Changes

- 新增檔案變更監聽流程，當已開啟專案中的關鍵檔案（`openspec/config.yaml`、`openspec/changes/**`、`openspec/specs/**`）更新時，系統即時重新載入資料。
- 在 UI 顯示即時更新狀態（例如「偵測到更新，已同步」或「同步失敗」）。
- 在重新整理資料時保留目前頁籤與選取項目，避免強制跳頁或中斷使用者閱讀。
- 讓本機可寫模式與 fallback 匯入模式都有一致的更新觸發與回饋邏輯（在 fallback 模式可採可行替代策略）。

## Capabilities

### New Capabilities
- `realtime-filesystem-refresh`: 提供已開啟專案檔案更新的即時偵測、資料重整與使用者回饋。

### Modified Capabilities
- 無（本變更以新增 capability 處理，不直接修改既有 main specs requirement）

## Impact

- 檔案存取層：需新增檔案變更事件監聽或輪詢策略與去抖動機制。
- 狀態管理層：需新增「同步中 / 同步完成 / 同步失敗」狀態與錯誤處理。
- 前端 UI：需新增即時更新提示元件，並確保更新後保留頁籤與選取上下文。
- 效能：需限制重複重整頻率，避免高頻檔案變動造成過度重算。
